# 0006: Detect monorepo root via topmost package.json, not Project.guessProjectDir

**Status:** Accepted
**Date:** 2026-05-04

`WorkspaceModelService` previously used `com.intellij.openapi.project.guessProjectDir(project)` (an extension on `Project`) at two call sites: `isPackageConfig` and `rootTurboFile`. That helper is deprecated in current platform versions, and CLAUDE.md is explicit that deprecated APIs should be flagged and replaced rather than carried forward. The semantic the plugin actually needs is narrower than "best-guess project base directory": both call sites want "the directory that hosts the Turborepo monorepo root", which by Turborepo's own contract is the directory containing the workspace-root `package.json` (the one with `workspaces` defined for npm/yarn/bun). Of the three replacement candidates considered (`ProjectRootManager.getInstance(project).contentRoots[0]`, `ProjectFileIndex` lookups, and the package.json topmost-path heuristic used by Qodana JS), the topmost-package.json heuristic is the one that aligns with what we actually mean: it deterministically picks the workspace root from `PackageJsonFileManager.validPackageJsonFiles` via `minByOrNull { it.path }?.parent`, has no dependency on how the user configured IDE content roots, and degrades gracefully (returns `null`) for projects with no resolvable `package.json`.

The replacement adds a private `monorepoRoot(): VirtualFile?` to `WorkspaceModelService` and substitutes it for both `project.guessProjectDir()` call sites. Edge cases worth documenting. (a) **No `package.json` in the project** → `monorepoRoot()` returns `null` and both `isPackageConfig` and `rootTurboFile` return `null`/`false`. This matches the prior `guessProjectDir`-returns-null branch and is the right semantic: if there is no `package.json`, there is no Turborepo workspace to inspect. (b) **Multiple `package.json` files at equal shortest path length** (i.e. same depth, different siblings) → `minByOrNull` picks one in iteration order. This is a degenerate setup that does not correspond to a valid Turborepo monorepo (a Turborepo project has exactly one root `package.json`); we accept the arbitrary choice rather than special-casing. (c) **Multi-content-root IDE projects with multiple monorepos** → the heuristic picks the topmost across the whole project, which may not be the user's "current" monorepo. We are deferring this rather than handling it now: it is a non-Turborepo-canonical setup, no current test fixture exercises it, and a fix would require either per-file enclosing-package detection (already on the v0.2 wishlist for env var leak detection) or per-content-root caching. The prior `guessProjectDir` behavior had the same limitation in different shape, so this is not a regression.
