# 0003 — `WorkspaceModelService` consolidates workspace-derived sets

**Status:** Accepted
**Date:** 2026-04-27

Inspection B (package-qualified task references) needs the set of known workspace package names; Inspection A (unresolved `dependsOn` references) already consumes the set of known task names from `KnownTaskNamesService`. Both sets share the same invalidation triggers — `PsiModificationTracker.MODIFICATION_COUNT` plus `PackageJsonFileManager.getInstance(project).modificationTracker` — and both are derived from the same file universe (`PackageJsonFileManager.validPackageJsonFiles` plus, for tasks, the root `turbo.json`). We renamed the service to `WorkspaceModelService` and exposed `taskNames: Set<String>` and `packageNames: Set<String>` as separate cached properties keyed on the `Project` user-data holder. The shared cache machinery (single service, single set of cache dependencies) means adding the package set required only a second computation function and a second `Key`, not a parallel service.

Alternative (b), introducing a sibling `KnownPackageNamesService` with the same shape, was rejected because the two sets would duplicate every cache dependency, lifecycle hook, and `getInstance` boilerplate while behaving identically — a parallel service would buy nothing except a longer name. Keeping the existing `KnownTaskNamesService` name was rejected because exposing `packageNames` from a class named after task names mis-describes the class's responsibility, and that mis-description compounds when v0.2 (run integration) and v0.4 (boundaries) add further workspace-derived state. The `PackageJsonData.getName()` accessor was rejected for the package-name reader in favor of PSI traversal of the top-level `name` property, matching how the same file's `scripts` are read — symmetry in the reader implementations is more valuable than the slightly shorter accessor call, and PSI traversal cannot silently disagree with `PackageJsonData`'s caching layer.
