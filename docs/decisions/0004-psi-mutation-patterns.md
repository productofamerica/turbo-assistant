# 0004 — Use `JsonPsiUtil.addProperty` to insert properties, not raw `addBefore`

**Status:** Accepted
**Date:** 2026-04-27

`AddExtendsFieldFix` for Inspection C (package-level `turbo.json` `extends` validator) needs to insert a new `"extends": ["//"]` property as the first key of the root JSON object. The natural-looking pattern — `JsonElementGenerator.createProperty(...)` followed by `rootObject.addBefore(newProperty, firstChild)` — produced syntactically invalid output: the inserted property had no trailing comma, so the result `{ "extends": [...] "tasks": {} }` was invalid JSON. This is a real correctness defect (mangled file), not a formatting nit; per the v0.1 guidance, structural insertions must round-trip through a valid grammar. The fix is to use `com.intellij.json.psi.JsonPsiUtil.addProperty(jsonObject, property, isFirst)` — a static helper shipped with the bundled JSON plugin (verified present in IDEA Ultimate 2025.2.6.1) that internally calls `addBefore`/`addAfter` and additionally inserts a `JsonElementGenerator.createComma()` element on the correct side of the new property. This is the canonical PSI-driven path for inserting a JSON property; any inspection or intention that adds a property to a `JsonObject` should prefer `JsonPsiUtil.addProperty` over hand-rolled `addBefore`/`addAfter` chains.

Alternatives considered. (a) Manually inserting a comma element via `JsonElementGenerator.createComma()` after the user's `addBefore` call — rejected because it duplicates exactly the work `JsonPsiUtil.addProperty` already does and risks subtle ordering bugs (e.g. inserting the comma before the property in the at-front case). (b) Falling back to `Document.replaceString` for the whole root object — rejected because it would bypass PSI invalidation and discard any author-formatted whitespace or comments in the file (relevant for `.jsonc`); the v0.1 guidance is explicit that `Document.replaceString` is reserved for content edits or real correctness escapes, not to paper over a syntactic defect that has a PSI-level fix. (c) Calling `CodeStyleManager.reformat(rootObject)` after the broken `addBefore` — rejected because the reformatter normalizes whitespace, not grammar; a missing comma stays missing.

See also: ADR 0005 documents a scoped carve-out where `Document.replaceString` is the chosen path for in-quotes content edits (`ReplaceWithNearMatchFix` in `DependsOnQuickFixes.kt`) because the canonical `ElementManipulators.handleContentChange` path triggers a JSON reformatter that materially changes the user's hand-authored array layout. That carve-out is consistent with this ADR's "content edits or real correctness escapes" reservation; structural edits still follow the patterns above.
