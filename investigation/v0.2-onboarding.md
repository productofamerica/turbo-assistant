## 3. Architectural decisions in force

`docs/decisions/`:

- **0001, JSON plugin dep**: `plugin.xml` declares hard `<depends>` on `JavaScript` and `com.intellij.modules.json`. Both are bundled in WebStorm and IntelliJ Ultimate; relying on bundled plugins is preferred over re-implementing JSON parsing. (Verified: `plugin.xml:13-15`, `build.gradle.kts:20-21`.)
- **0002, JSONC support via `language="JSON"`**: a single `language="JSON"` registration covers `.jsonc` because the bundled JSON plugin parses both into the same PSI hierarchy. (Verified: every `<localInspection>` in `plugin.xml` uses `language="JSON"`; `util/TurboFiles.kt:5-6` accepts both filenames; `testJsoncUnknownTaskWarns` exercises a `.jsonc` file with comments.)
- **0003, `WorkspaceModelService` (renamed from `KnownTaskNamesService`)**: a single project-level service exposing `taskNames`, `packageNames`, `isPackageConfig`, `rootTurboFile`, `packageJsonPsiFiles`, `findPackageJsonByName`, sharing one cache invalidation policy (PSI modification + `PackageJsonFileManager.modificationTracker`). PSI traversal preferred over `PackageJsonData` because `getScripts()` does not exist on that class.
- **0004, PSI mutation patterns**: `JsonPsiUtil.addProperty` for inserting a property (avoids manual comma handling); `JsonElementGenerator.createValue<JsonArray>` plus `.replace()` for shape changes; `WriteCommandAction.runWriteCommandAction` to wrap the mutation. (Verified: `PackageConfigExtendsQuickFixes.kt:14-43` uses both patterns.) Note: `DependsOnQuickFixes.kt:23-28` (`ReplaceWithNearMatchFix`) deviates from the "ElementManipulators for content" intent of this ADR by using `document.replaceString`. Either the ADR needs a sub-clause about content edits, or this quick fix needs to be ported to `ElementManipulators.handleContentChange`.

(`DependsOnUndefinedTaskInspectionTest.kt:5-12` is the template.)
- **`testHighlighting(true, false, true)` plus `<warning descr="...">`-style markers in fixture files**. Verified by `myFixture.testHighlighting(true, false, true)` calls in every inspection test (e.g. `DependsOnUndefinedTaskInspectionTest.kt:16, 21, ...`). Each quick fix test loads the fixture, invokes `myFixture.testHighlighting`, calls a private `moveCaretInsideStringLiteral(needle)` or `moveCaretTo(needle, offset)` helper, then `myFixture.findSingleIntention(actionDescription)`, `myFixture.launchAction(intention)`, and finally `myFixture.checkResultByFile("turbo.json", "<case>.after/turbo.json", true)`. ## 5. v0.2 problem space (understanding only, not a design)

**This section assumes the user wants v0.2 reordered to be env var leak detection.** That assumption is contradicted by the roadmap (see section 8, open question 1). If the assumption is wrong, ignore this section and treat v0.2 as run integration per `docs/roadmap.md:13-19`.

Env var leak detection is an inspection that flags `process.env.X` (and equivalent) reads in JS/TS source where `X` is not declared in any applicable `turbo.json` env config. The motivation is that Turborepo only re-runs cached tasks when declared inputs change. Reading an undeclared env var inside a task means cache hits will silently return stale output keyed on a different env. It is "Turborepo's #1 caching footgun" per `docs/roadmap.md:23`.

**Detection patterns to support** (from Vercel's `eslint-plugin-turbo`, `no-undeclared-env-vars` rule):

- `process.env.X` (member access via identifier).
- `process.env["X"]` and `process.env['X']` (computed access via string literal).
- `import.meta.env.X` (Vite-style).
- Destructuring: `const { X, Y } = process.env;` and equivalent reassignments.

**Validation sources**:

- User-defined allowList of regex patterns (settings panel; persisted via `PersistentStateComponent`).
- Framework-inferred patterns based on the file's nearest `package.json` dependencies, e.g. `NEXT_PUBLIC_*` for Next.js, `VITE_*` for Vite. Vendor `frameworks.json` from `@turbo/types`.
- Per-file applicability: every `turbo.json` whose env config could apply (root `globalEnv`, root `globalPassThroughEnv`, root `globalDotEnv`, legacy `$VAR` entries in `dependsOn`, plus per-task `env`/`passThroughEnv`/`dotEnv` reachable via the package the file belongs to).

**Explicitly out of scope** (matches eslint-plugin-turbo behavior):

- Dynamic access where the variable name is itself a variable: `process.env[varName]`.
- Diffing committed `.env.*` files against declared globalDotEnv.
- Following imports across packages to detect transitive env reads.
- Refactoring (renaming env vars, etc.).

## 7. What v0.2 needs that v0.1 does not have

- **JS/TS PSI knowledge**. The current research files cover JSON PSI thoroughly; `research/json-psi.md` and `research/workspace-model.md` say nothing about JS PSI. The JavaScript plugin's PSI is in the proprietary plugin (closed source). Open-source consumers of it (`JetBrains/intellij-plugins` repo, e.g. `prettierJS`, Vue, Astro) are the only public references. v0.2 needs:
  - The visitor base class for JS expressions (likely `JSElementVisitor` or `JSRecursiveElementVisitor`).
  - The PSI types for `process.env.X` (member expressions) and `process.env["X"]` (indexed access expressions).
  - The PSI shape of destructuring patterns (`JSDestructuringObject`, etc.).
  - The visitor pattern for `import.meta.env.X` and how it differs from `process.env`.
  - The right way to gate inspection registration to JS and TS only (`language="JavaScript"`, `language="TypeScript"`, possibly `language="ECMAScript 6"` for older platforms).
- **A wildcard-to-regex utility**. Port `eslint-plugin-turbo`'s `lib/utils/wildcard-processing.ts`. Translates patterns like `NEXT_PUBLIC_*` to `^NEXT_PUBLIC_.*$`. The implementation has corner cases (escaping regex metacharacters, behavior on `**`, behavior on patterns containing `?`).
- **A framework registry**. Vendor `frameworks.json` from the `@turbo/types` package. Map: framework name to glob env wildcards. Choose between (a) build-time fetch and freeze (reproducible builds, requires updates), (b) hardcode (simplest, requires updates), (c) read from `node_modules/@turbo/types/...` at runtime (most flexible, fragile if `node_modules` is absent or stale). Recommend (b) for v0.2 with a doc note explaining the trade-off.
- **A user-facing settings panel**. Likely under `Settings > Tools > Turbo Assistant`. Fields: an editable list of regex patterns (allowList) and a toggle to enable framework auto-detection. Implemented via `Configurable` and a Kotlin Swing UI form (or DSL), with state persisted via `PersistentStateComponent`.
- **A `PersistentStateComponent`** (project-level or application-level; project-level is more typical for repo-specific settings). Stores `allowList: List<String>` and `frameworkAutoDetect: Boolean`.

## 9. Pitfalls flagged for v0.2

- **Performance**. Every keystroke in a JS/TS file fires the inspection. `eslint-plugin-turbo` runs in a separate process and uses MD5 hashing of `turbo.json` content to invalidate; the IDE has cheap PSI modification trackers, so the hash machinery is unnecessary. The Project-style cache must be `getProjectPsiDependentCache(jsFile)` keyed off the JS file's PSI, with `WorkspaceModelService` providing the env declarations as already-cached `Set<String>` lookups.
- **Closed-source JS PSI**. The JavaScript plugin is proprietary. Research has to lean on `JetBrains/intellij-plugins` open-source consumers (Prettier, Vue, Astro). `JSReferenceExpression` and `JSElementVisitor` are the canonical names but exact class paths and method signatures should be verified by reading those open-source plugins, not invented.
- **Destructuring is a separate visitor branch**. `const { FOO } = process.env` is not a `JSReferenceExpression` of `process.env.FOO`; it is a destructuring binding. The visitor needs to detect destructuring patterns where the source is `process.env` and walk the destructured names. Same for `import.meta.env`. Easy to miss; needs an explicit fixture.
- **`import.meta.env` differs from `process.env` in PSI shape** (it is a meta-property expression, not a member expression). A single visitor that pattern-matches on "two-level member access starting with a known root" will miss `import.meta.env`.
- **Computed access with non-literal subscript** (`process.env[someVar]`) must be skipped, not flagged. Only literal subscripts count as detected reads.
- **`NEXT_PUBLIC_*` and `VITE_*` are framework-specific**. The framework registry has to be applied per-file (based on the file's nearest `package.json` deps), not project-wide. Two packages with different frameworks in the same monorepo need different validation.
- **Multi-resolve for package boundaries**. A file in `apps/web/` may have its env declaration in `apps/web/turbo.json` (via `extends: ["//"]`) plus `turbo.json` at root. Resolving "is this env var declared anywhere applicable?" walks the inheritance chain.
